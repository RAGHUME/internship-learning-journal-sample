# Network Visualization

## Procedure: Actor Network Extraction for Kumu

### 1. Data Preparation
* **Source:** Download `title.basics.tsv.gz`, `title.principals.tsv.gz`, and `name.basics.tsv.gz` from IMDb.
* **Filter (Principals):** Keep only rows where `category` is `actor` or `actress`.
* **Filter (Titles):** Keep only rows where `titleType` is `movie`.

### 2. Matrix Construction
Convert the filtered DataFrame into a **Sparse Matrix** (CSR format) to handle the large volume of data efficiently.
* **Rows:** Movies (`tconst`)
* **Columns:** Actors (`nconst`)
* **Values:** `1` (indicates presence in the movie)

### 3. Co-occurrence Calculation
Use linear algebra to find shared credits between all actors:
1.  **Multiply:** $A^T \times A$ (Transpose of Movie-Actor matrix multiplied by itself).
2.  **Clear Self-Loops:** Set the diagonal of the resulting matrix to `0` (an actor cannot be paired with themselves).
3.  **Coordinate Format:** Convert the result to `COO` format to extract `row` (Actor A), `col` (Actor B), and `data` (Weight/Count).

### 4. Thresholding & Export
* **Filter Pairs:** Set a minimum threshold (e.g., `min_pairs=3`) to remove weak or accidental connections.
* **Map Names:** Join the results with the Names dataset to replace IDs with human-readable names.
* **Save:** Export the final DataFrame to `.xlsx` or `.csv`.

### 5. Kumu Visualization
1.  Open [Kumu.io](https://kumu.io).
2.  Create a new project and select **Stakeholder** or **Network** template.
3.  Upload the exported file.
4.  **Mapping:** * Set `name1` and `name2` as the **From** and **To** fields (Elements).
    * Set `count` as the **Connection Strength**.

# AI Visualization :
## From prompt to plot

### Step 1: Finding a Dataset
* **Brainstorming**: Rather than manually searching, use conversational voice prompting with an LLM to brainstorm dataset ideas based on your personal interests.
* **Selection**: In this workshop, a **Goodreads 100k dataset** (containing book formats, pages, ratings, and reviews) was selected and downloaded from Kaggle.

### Step 2: Ideating Stories
* **Efficiency**: To save time and computational resources, do not immediately upload the entire dataset.
* **Metadata First**: Copy only the **column names and metadata** and paste them into the LLM.
* **Strategy**: Ask the LLM to generate multiple interesting "data stories," identify target audiences, and propose visualization strategies.
* **Example Hypotheses**: 
    * Do longer books score higher ratings? 
    * Is there a difference between hardcovers and paperbacks?

### Step 3: Analyzing the Data
* **Code Interpretation**: Upload the full dataset to an LLM with a code interpreter.
* **Statistical Rigor**: Instruct the LLM to run statistical hypothesis tests, drop statistically insignificant results, and generate initial exploratory charts (e.g., **Matplotlib scatter plots**).
* **Domain Expertise Check**: Review the LLM's analysis to ensure it hasn't hallucinated.
    * **Note**: Human judgment is required to determine if a finding is **practically significant** (e.g., an LLM might find that 100 extra pages increase a rating by only **0.06**, which is mathematically real but functionally tiny).

### Step 4: Visualizing and Publishing
* **AI Coding Agents**: Use agents to build and deploy the web page instead of manual coding.
* **Setup**: Create a **GitHub repository** and upload your compressed dataset.
* **Prompting**: Ask an agent (like Jules or CodeX) to write **HTML, CSS, and JavaScript**.
* **Optimization**: Instruct the LLM to **randomly sample data** for visual scatter plots (to ensure fast loading) while keeping statistical calculations based on the full dataset.
* **Deployment**: Use GitHub pull requests generated by the agents to deploy via **GitHub Pages**, turning code into a live website.

### Step 5: Debugging via Context Engineering
* **Definition**: Instead of fixing code yourself, use **Context Engineering**—passing the exact error information back to the LLM so it can fix its own mistakes.
* **Process**: 
    1.  Open browser **Developer Tools** ($F12$ or $Ctrl+Shift+I$).
    2.  Go to the **Console** tab.
    3.  Copy red error messages and paste them back into the LLM with a prompt like: *"I got this error, fix it."*

## Key Philosophies for the AI Era

> ### Resist the Urge to Do Manual Work
> Even if you have 30 years of coding experience, train yourself to let the LLM solve the problem. Manual fixes do not scale; building workflows where the LLM does the work allows you to achieve exponentially more.

* **Know When to Give Up (or Wait)**: Given the "jagged edge" of AI, if an LLM is hopelessly stuck, it is often better to drop the project or wait a few months for a more capable model.
* **Mitigate Bias**: Beware of "narrative biases" where you only pursue data that confirms your beliefs. Use your personal experience to spot when an analysis "doesn't look right."


# Data Storytelling with LLMs:

## 1. The Core Workflow: From Prompt to Plot

### Step 1: Finding a Dataset
* **Method**: Use conversational voice prompting with an LLM to brainstorm dataset ideas based on personal interests.
* **Example**: Selecting a "Goodreads 100k" dataset from Kaggle to explore book trends.

### Step 2: Ideating Stories
* **Strategy**: Do not upload the entire dataset immediately. Instead, paste only the **column names and metadata**.
* **Goal**: Ask the LLM to generate multiple "data stories," identify target audiences, and propose visualization strategies.
* **Hypothesis Generation**: Formulate questions like "Do longer books score higher ratings?" or "How do formats affect popularity?"

### Step 3: Analyzing the Data
* **Code Interpretation**: Upload the full dataset and instruct the LLM to run statistical tests (e.g., hypothesis testing) and generate exploratory charts (e.g., Matplotlib).
* **Human Oversight**: Review the analysis to ensure the LLM hasn't "hallucinated" data. 
* **Practical Significance**: Distinguish between "mathematically real" effects and "practically significant" ones (e.g., a tiny rating increase over hundreds of pages).

### Step 4: Visualizing and Publishing
* **Coding Agents**: Use AI agents (like Jules or CodeX) to write HTML, CSS, and JavaScript.
* **Optimization**: Instruct the LLM to randomly sample data for the front-end visual (to ensure fast loading) while keeping the heavy calculations based on the full dataset.
* **Deployment**: Use GitHub Pages to turn the code into a live, interactive website.

### Step 5: Debugging (Context Engineering)
* **Process**: Instead of manual fixing, pass error messages back to the LLM.
* **Workflow**: Open Browser DevTools (F12) -> Copy Console Errors -> Paste to LLM with the prompt: "I got this error, fix it."

## 2. Technical Tactics: Data Design by Dialogue

### Data Extraction
* **Dialogue Scraping**: Ask an LLM to write a JavaScript snippet for the browser console to extract data from non-exportable sources (like WhatsApp Web).

### Topic Modeling & Clustering
* **Automated Scripting**: Use agents to write Python scripts (e.g., `topics.py`) that:
    1.  Calculate **Embeddings** for text.
    2.  Cluster data using **K-Means**.
    3.  Use the LLM to **name the clusters** (e.g., "Social Media Updates," "Admin Activity").

### The "Impossibility List"
* Maintain a list of tasks the LLM fails at today (e.g., perfectly applying complex brand style guides to charts).
* Revisit this list monthly; as models improve, "impossible" tasks often become automated.

## 3. Key Philosophies for the AI Era

### Resist Manual Work
* Even if you are an expert coder, prioritize letting the LLM solve the problem. Manual fixes do not scale; building LLM-driven workflows does.

### The "Jagged Edge"
* LLMs are postgraduate-level in some areas and grade-school level in others. Constant experimentation is required to find where the "edge" lies for your specific task.

### Multi-Output Strategy
* Never ask for just one result. Always ask for **10 diverse data stories** or 10 versions of a chart. This mitigates bias and increases the chance of finding a unique insight.

### Context over Coding
* The primary skill of a modern data professional is shifting from **syntax** (how to write code) to **context** (knowing what to ask and how to verify the significance of the result).
